# 需求清单与边界条件

## 1. 功能需求（二选一或双做）

### A. 揭棋对弈程序（网络）

- [F1] 两个客户端经 TCP 连接同一服务器真人对弈  
- [F2] 服务器与客户端均校验并**拒绝非法着法**  
- [F3] 不考虑「不应将」自动判负；走不出不自动判胜  
- [F4] 棋谱：坐标 9-0 行、a-i 列；每步 source + destination；首翻带 type  
- [F5] 服务器随机决定翻开类型；自动记录棋谱  
- [F6] `Move` 含 source、destination、type、时间戳；**以服务器时间为准**  
- [F7] 计时器判超时（可加网络延迟裕量，如 +5s）  
- [F8] 客户端 UI 从简；可选多盘对弈  

### B. AI 博弈

- [F9] 暗子局面评估含**数学期望值**  
- [F10] Alpha-Beta 剪枝搜索  
- [F11] Agent 对象封装决策  
- [F12] 可对接网络或本地 Board 自检  

## 2. 规则需求（揭棋）

- 开局：仅将帅明，其余 15 子暗子随机位置  
- 走暗子按**位置角色**规则；走完或吃子后翻开；可原地翻子占一回合  
- 明士、明象可过河；塞象眼、蹩马腿同象棋  
- 胜负：将死、困毙、认输、超时  
- 和棋：连续 40 步无吃子；长将/长捉（兵卒长捉和）  

## 3. 非功能需求

- 面向对象：≥5 领域类  
- 组间互操作：协议文档化（`INTERFACE.md`）  
- 必须使用 AI 辅助分析、设计、编程、测试（报告说明过程）  
- 实验报告：成员分工 + 贡献度百分比  

## 4. 边界与不做项

- 不实现精美 GUI（可选简单控制台）  
- 不处理客户端伪造时间戳（服务器忽略）  
- 「不应将」导致的直接吃帅赢：**不**由系统自动判定
